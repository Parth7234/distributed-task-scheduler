# Testing Distributed Task Scheduler for Docker

A step-by-step guide to test all 3 components — API, Worker, and Sweeper.

Step 0 — Start Everything

cd distributed-task-scheduler
docker compose up --build -d

The `-d` flag runs everything in the background so the terminal stays free. We can always check logs with `docker compose logs -f`.

Step 1 — Post Tasks via the API

Submit a few tasks with different priorities (lower number = higher priority):

# High priority
curl -X POST http://localhost:8000/tasks \
  -H "Content-Type: application/json" \
  -d '{"priority": 1, "description": "https://images.unsplash.com/photo-1503023345310-bd7c1de61c7d"}'

# Medium priority
curl -X POST http://localhost:8000/tasks \
  -H "Content-Type: application/json" \
  -d '{"priority": 5, "description": "https://images.unsplash.com/photo-1550684848-fac1c5b4e853?q=80&w=3000&auto=format&fit=crop"}'

# Low priority
curl -X POST http://localhost:8000/tasks \
  -H "Content-Type: application/json" \
  -d '{"priority": 10, "description": "sweeper_testing"}'

Each response will confirm the task was created with status `PENDING`.

Step 2 — Watch the Worker Execute Tasks

Open the worker logs to see it pick up tasks in priority order:
docker compose logs -f worker

You'll see output like:

[Worker] Grabbed Task ID: 1 (Priority: 1.0)
[Worker] Executing: 'Process payment'...
[Worker] Successfully finished Task ID: 1
[Worker] Grabbed Task ID: 2 (Priority: 5.0)

Step 3 — Check Task Status via API

curl http://localhost:8000/tasks

You'll see all tasks with their current status:
- `PENDING` → waiting in queue
- `RUNNING` → worker is currently executing it
- `COMPLETED` → done

We can also use the Swagger UI at **http://localhost:8000/docs** to submit and view tasks without using `curl`.

Testing the Sweeper (Crash Recovery)

This is the most interesting part. The **Sweeper** automatically detects tasks that have been `RUNNING` for more than 60 seconds (meaning the worker probably crashed) and requeues them.

How to simulate a crash:

Submit a new task:

curl -X POST http://localhost:8000/tasks \
  -H "Content-Type: application/json" \
  -d '{"priority": 10, "description": "sweeper_testing"}'

Watch the worker log until it says `Executing:...`:
docker compose logs -f worker

Immediately kill the worker container (simulating a crash):
docker compose kill worker

This kills the worker mid-execution. The task is stuck as `RUNNING` in the database — it will never finish on its own.

Verify the task is stuck:
curl http://localhost:8000/tasks
You'll see the crash test task with `"status": "RUNNING"` — it's a ghost task now.

Watch the sweeper rescue it — check sweeper logs:
docker compose logs -f sweeper

After **~60 seconds**, the sweeper will detect the ghost task and print:
[Sweeper] Found crashed Task ID: X. Requeuing...
[Sweeper] Successfully rescued tasks.

Restart the worker to process the rescued task:
docker compose start worker

Verify the task completed:
curl http://localhost:8000/tasks
The crash test task should now show `"status": "COMPLETED"` 


## Useful Docker Commands

| Command | What it does |
|---|---|
| `docker compose up --build -d` | Start everything in background |
| `docker compose down` | Stop and remove all containers |
| `docker compose ps` | See which containers are running |
| `docker compose logs -f` | Follow all logs live |
| `docker compose logs -f worker` | Follow only the worker's logs |
| `docker compose kill worker` | Force-kill the worker (simulate crash) |
| `docker compose start worker` | Restart a stopped service |
| `docker compose exec worker ls /app/thumbnails/` | List thumbnails inside the container |
| `docker compose cp worker:/app/thumbnails/ ./thumbnails_local/` | Copy them to your local machine |
